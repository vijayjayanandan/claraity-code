# Codebase Survey — Implementation Plan

**Status:** Parked (Phase 2 — after Director Adapter is integrated)

## Context

The Director Protocol MVP is built (130 tests, all green). But the Director has no memory — every session starts from scratch with zero knowledge of the codebase. Before the Director can work effectively, it needs a persistent understanding of the project.

This plan is parked until the Director Adapter (Phase 1) wires the protocol into the CodingAgent. The survey is Phase 2 — it gives the adapter persistent context to load.

### Two Scenarios

**Existing Project:** Before the agent touches any code, it surveys the entire codebase and creates a persistent `codebase_context.md`. Every future session loads this document first. The agent refuses to work without it.

**Greenfield Project:** The agent first understands what it's building, documents the vision and architecture. This becomes the persistent context. Built incrementally — MVP first, then roadmap.

## Goal

Build a codebase surveyor that scans the project and generates a persistent `codebase_context.md`. This document is loaded into every future session, giving the agent instant project knowledge without re-exploring.

**Trigger:** User types `/code-survey` in TUI, or agent offers to survey on first session when no context exists.

**Depth:** File-level scan with brief descriptions of what each code file does (from module docstrings).

**Storage decision:** Markdown-first. The LLM consumes text directly — no serialization step. Human-readable, version-controllable. SQLite backing store added later when we need queryability.

## Vertical Slices

### Slice 1: Survey Data Models

**Analogy:** The survey forms — blank templates for recording what the scanner finds.

**Create:**
- `src/director/survey/__init__.py`
- `src/director/survey/models.py`
- `tests/director/survey/__init__.py`
- `tests/director/survey/test_models.py`

**What's in it:**

| Class | Purpose |
|-------|---------|
| `FileInfo` | One file: path, extension, line_count, size_bytes, description |
| `ModuleInfo` | One directory: path, description, list of FileInfo |
| `ProjectInfo` | Project-level: name, language, frameworks, entry_points, test_framework |
| `SurveyMetadata` | When/what: timestamp, git_commit, total_files, total_lines |
| `SurveyResult` | Everything: metadata + project + list of modules |

**Test criteria:**
- All dataclasses create with required fields
- Defaults work (empty lists, None)
- `SurveyResult.to_dict()` serializes correctly
- `SurveyMetadata` stores timestamp and git commit

### Slice 2: File Scanner

**Analogy:** The surveyor walking every hallway of the building, noting each room's name, size, and door plaque (docstring).

**Create:**
- `src/director/survey/scanner.py`
- `tests/director/survey/test_scanner.py`

**What's in it:**
- `CodebaseScanner` class
  - `scan(root_path)` -> list of FileInfo
  - `_should_skip(path)` -> respects exclusion patterns
  - `_count_lines(path)` -> line count
  - `_extract_docstring(path)` -> reads Python module docstring using `ast.parse`
  - `_group_into_modules(files)` -> groups FileInfo by parent directory into ModuleInfo

**Exclusion patterns (hardcoded for MVP):**
- `.git/`, `__pycache__/`, `.venv/`, `venv/`, `node_modules/`
- `.eggs/`, `build/`, `dist/`, `.mypy_cache/`, `.pytest_cache/`
- `.clarity/` (agent workspace — don't index ourselves)
- Binary extensions: `.pyc`, `.so`, `.db`, `.sqlite`, `.png`, `.jpg`

**Test criteria:**
- Scans a temp directory with known structure
- Skips excluded directories
- Counts lines correctly
- Extracts Python docstrings (module-level)
- Returns empty string for files without docstrings
- Groups files into modules by directory
- Handles non-Python files (counts lines, no docstring)

---

### Slice 3: Project Detector

**Analogy:** Reading the building's title deed — what's this project, what's it built with, where's the front door?

**Create:**
- `src/director/survey/project_detector.py`
- `tests/director/survey/test_project_detector.py`

**What's in it:**
- `ProjectDetector` class
  - `detect(root_path)` -> ProjectInfo
  - `_detect_from_pyproject(path)` -> reads pyproject.toml
  - `_detect_entry_points(root_path)` -> finds main/cli/app files
  - `_detect_test_framework(root_path)` -> pytest/unittest/etc.

**Detection logic:**
- Name: from pyproject.toml `[project].name`
- Language: Python (from .py files), JS/TS (from .js/.ts files)
- Frameworks: from dependencies in pyproject.toml (textual, flask, fastapi, etc.)
- Entry points: from `[project.scripts]` or common patterns (cli.py, main.py, app.py)
- Test framework: from pytest in dev deps, or presence of conftest.py

**Test criteria:**
- Detects from a minimal pyproject.toml
- Handles missing pyproject.toml gracefully
- Finds entry points from scripts section
- Detects pytest as test framework
- Returns sensible defaults when detection fails

### Slice 4: Markdown Generator

**Analogy:** The report writer — takes the raw survey data and produces the final document.

**Create:**
- `src/director/survey/generator.py`
- `tests/director/survey/test_generator.py`

**What's in it:**
- `ContextGenerator` class
  - `generate(survey_result)` -> markdown string
  - `_render_header(metadata, project)` -> title + overview
  - `_render_structure(modules)` -> directory tree with annotations
  - `_render_module_catalog(modules)` -> per-module tables
  - `_render_key_files(modules)` -> top N files by line count
  - `_render_metadata(metadata)` -> survey footer

**Test criteria:**
- Generates valid markdown from a SurveyResult
- Header includes project name, timestamp, git commit
- Structure section shows directory tree
- Module catalog shows files with line counts and descriptions
- Key files sorted by line count descending (top 15)
- Empty modules are skipped
- Output is a string (no file I/O in generator — manager writes)

---

### Slice 5: Survey Manager + Staleness Detection

**Analogy:** The project manager who schedules the survey, checks if the building has changed since last inspection, and files the report.

**Create:**
- `src/director/survey/manager.py`
- `tests/director/survey/test_manager.py`

**Modify:**
- `src/director/survey/__init__.py` (final exports)

**What's in it:**
- `SurveyManager` class
  - `survey(root_path)` -> SurveyResult (orchestrates scan + detect + generate)
  - `save(root_path, result)` -> writes .clarity/codebase_context.md
  - `load(root_path)` -> reads existing context (returns None if missing)
  - `is_stale(root_path)` -> checks git diff since last survey commit
  - `_get_current_git_commit(root_path)` -> current HEAD short hash
  - `_parse_survey_commit(content)` -> extract commit hash from existing markdown header

**Staleness logic:**
- Parse the git commit from the existing codebase_context.md header
- Compare against current HEAD
- If different, the survey is stale
- If no context file exists, survey is needed

**Test criteria:**
- `survey()` produces a SurveyResult with all fields populated
- `save()` writes markdown to .clarity/codebase_context.md
- `load()` returns content when file exists, None when missing
- `is_stale()` returns True when git commit differs
- `is_stale()` returns True when no context file exists
- `_get_current_git_commit()` returns short hash string
- Full round-trip: survey -> save -> load -> content matches

## Patterns to Follow

- **Logging:** `from src.observability import get_logger` (no raw logging)
- **No emojis** in Python code
- **Dataclasses** for all data structures
- **pathlib.Path** for all file operations
- **ast module** for Python docstring extraction (no execution)
- **Exclusion patterns** hardcoded (no external dependency for MVP)
