# Plan: Replace Jest with Vitest in claraity-vscode

## Summary

Migrate the `claraity-vscode` extension's test infrastructure from Jest + ts-jest to Vitest.
The `webview-ui` sub-package already uses Vitest -- this migration makes the entire TypeScript
codebase consistent.

## Scope

Only `claraity-vscode/` is affected. `webview-ui/` already uses Vitest and is untouched.

## What Exists Today

| File | Role |
|------|------|
| `claraity-vscode/jest.config.js` | Jest config: ts-jest preset, node env, `__mocks__/vscode.js` alias, coverage |
| `claraity-vscode/package.json` | `"test": "jest"`, `"test:coverage": "jest --coverage"`, devDeps: `jest`, `@types/jest`, `ts-jest` |
| `claraity-vscode/__mocks__/vscode.js` | Manual vscode mock using `jest.fn()` |
| `claraity-vscode/__mocks__/ws.js` | Manual ws mock (plain EventEmitter, no jest APIs) |
| `claraity-vscode/src/__tests__/*.test.ts` | 8 test files using `jest.mock()`, `jest.fn()`, `jest.Mock` type, `jest.clearAllMocks()`, `jest.useFakeTimers()` |

## Key Compatibility Notes

Vitest is API-compatible with Jest for the patterns used here:
- `vi.mock()` replaces `jest.mock()` -- same call signature
- `vi.fn()` replaces `jest.fn()` -- identical API
- `vi.Mock` replaces `jest.Mock` type
- `vi.clearAllMocks()`, `vi.useFakeTimers()`, `vi.useRealTimers()`, `vi.advanceTimersByTime()` -- all exist in Vitest
- `vi.mocked()` replaces casting `as jest.Mock` (optional but cleaner)
- `beforeEach`, `describe`, `test`, `expect` -- unchanged globals

The `__mocks__/vscode.js` file uses `jest.fn()` extensively. In Vitest, `vi.fn()` is the
equivalent. Since Vitest provides globals (`globals: true`), `vi` is available globally.
However, the mock file uses CommonJS `module.exports` and `jest.fn()` directly. We need to
update it to use `vi.fn()` instead.

The `__mocks__/ws.js` file uses only Node's `EventEmitter` -- no Jest APIs -- so it needs
no changes.

## Changes Required

### 1. `claraity-vscode/package.json`
- Remove: `jest`, `@types/jest`, `ts-jest` from `devDependencies`
- Add: `vitest` to `devDependencies`
- Update scripts:
  - `"test": "vitest run"` (was `"jest"`)
  - `"test:coverage": "vitest run --coverage"` (was `"jest --coverage"`)

### 2. `claraity-vscode/jest.config.js` -- DELETE
- No longer needed. Vitest config goes in `vitest.config.ts`.

### 3. `claraity-vscode/vitest.config.ts` -- CREATE NEW
```ts
import { defineConfig } from 'vitest/config';

export default defineConfig({
    test: {
        environment: 'node',
        globals: true,
        include: ['src/**/__tests__/**/*.test.ts'],
        alias: {
            vscode: new URL('./__mocks__/vscode.js', import.meta.url).pathname,
        },
        clearMocks: true,
        restoreMocks: true,
        coverage: {
            include: ['src/**/*.ts'],
            exclude: ['src/**/__tests__/**'],
            reporter: ['text', 'text-summary'],
            reportsDirectory: 'coverage',
        },
    },
});
```

Note: The `moduleNameMapper` in jest.config.js mapped `^vscode$` to `__mocks__/vscode.js`.
Vitest uses `test.alias` for the same purpose.

### 4. `claraity-vscode/__mocks__/vscode.js` -- UPDATE
Replace all `jest.fn(...)` calls with `vi.fn(...)`. Since Vitest globals are enabled,
`vi` is available without import. The file uses CommonJS `module.exports` which Vitest
supports fine.

### 5. Test files -- UPDATE (8 files)
Replace Jest-specific APIs with Vitest equivalents:
- `jest.mock(...)` -> `vi.mock(...)`
- `jest.fn()` -> `vi.fn()`
- `jest.Mock` type -> `vi.Mock` (or `ReturnType<typeof vi.fn>`)
- `jest.clearAllMocks()` -> `vi.clearAllMocks()`
- `jest.useFakeTimers(...)` -> `vi.useFakeTimers(...)`
- `jest.useRealTimers()` -> `vi.useRealTimers()`
- `jest.advanceTimersByTime(n)` -> `vi.advanceTimersByTime(n)`
- Add `import { vi } from 'vitest'` to each test file that uses `vi.*`
  (since globals:true covers `describe/test/expect` but `vi` still needs explicit import
  OR we rely on the global -- Vitest exposes `vi` as a global when globals:true)

Actually with `globals: true`, `vi` IS a global in Vitest. So no import needed.
But TypeScript needs the type. We add `"types": ["vitest/globals"]` to tsconfig,
OR add `/// <reference types="vitest/globals" />` to each file,
OR simply import `{ vi }` from `'vitest'` in each file.

The cleanest approach: add `"vitest/globals"` to `tsconfig.json` `compilerOptions.types`.

### 6. `claraity-vscode/tsconfig.json` -- UPDATE
Add `"types": ["vitest/globals"]` to `compilerOptions` so TypeScript recognizes
`vi`, `describe`, `test`, `expect`, etc. as globals without explicit imports.

## Execution Order

1. Update `package.json` (remove Jest deps, add Vitest, update scripts)
2. Delete `jest.config.js`
3. Create `vitest.config.ts`
4. Update `__mocks__/vscode.js` (jest.fn -> vi.fn)
5. Update all 8 test files (jest.* -> vi.*)
6. Update `tsconfig.json` (add vitest/globals types)
7. Run `npm install` in `claraity-vscode/`
8. Run `npm test` to verify

## Risks

- `ts-jest` compiled TypeScript on-the-fly; Vitest uses esbuild/vite for TS transpilation.
  This is faster and should be transparent for these tests.
- The `module: "commonjs"` in tsconfig.json is fine -- Vitest handles this.
- Windows path handling in the `alias` config: use a relative string path instead of
  `import.meta.url` URL construction to avoid Windows path issues.
