# Plan: Surface Config Errors from LLM Failures

## Problem

When an LLM call fails due to config issues (bad API key, invalid model, bad max_tokens / temperature), the user sees a generic "Connection error. Please check your network." message. The actual error from OpenAI (e.g. `AuthenticationError: Incorrect API key provided`) is only visible in the logs.

## Root Cause

In `src/core/agent.py` lines 2064-2069, the error classification is binary:
- timeout -> "Request timed out..."
- everything else -> "Connection error. Please check your network." with `error_type: "network"`

Config-related API errors (401 auth, 400 bad request with invalid params, 404 model not found) all fall into the "everything else" bucket and get the same generic message.

## What Already Works (No Changes Needed)

- `ErrorEvent` already serializes all fields via `asdict()` in `serializers.py` -- `error_type` and `recoverable` are already sent over the wire
- `dispatch.ts` already passes `errorType: msg.error_type` into the `ERROR` action
- `actions.ts` already has `ERROR` with `errorType: string`
- `sidebar-provider.ts` already disconnects on `recoverable === false` (line 307-309)

## Solution

### 1. Python: Better error classification in `src/core/agent.py`

Replace the two-branch classification (timeout / non-timeout) with a three-branch version. Inspect `error_msg` (the wrapped RuntimeError message from `openai_backend.py`) AND `root_cause_message` for config error signals:

**Auth/API key errors**: "authentication", "api key", "invalid_api_key", "401", "incorrect api key", "unauthorized"
**Invalid model**: "model not found", "invalid_model", "no model", "does not exist"  
**Bad parameters**: "invalid_request_error", "max_tokens", "temperature", "top_p", "invalid value for", "must be between"

For config errors:
- Extract the provider's actual error sentence from `root_cause_message` (first 200 chars, already sanitized by the failure handler)
- Build a specific `user_message` like: `"API configuration error: <provider message>. Open Settings to fix."`
- Emit `ErrorEvent(error_type="config_error", recoverable=False, ...)`

### 2. Webview state: Add `errorType?` to error TimelineEntry

**`claraity-vscode/webview-ui/src/state/state.ts`** line 35:
```ts
// Before:

// After:

```

### 3. Webview reducer: Pass errorType into timeline entry

**`claraity-vscode/webview-ui/src/state/reducer.ts`** in the `ERROR` case:
```ts
// Before:
{ type: "error" as const, id: errId, message: action.message }
// After:
{ type: "error" as const, id: errId, message: action.message, errorType: action.errorType }
```

### 4. ChatHistory: Add `onOpenSettings` callback prop

**`claraity-vscode/webview-ui/src/components/ChatHistory.tsx`**:
- Add `onOpenSettings?: () => void` to `ChatHistoryProps`
- In the `error` case of the timeline map, render an "Open Settings" button when `entry.errorType === "config_error"`:

```tsx
case "error":
  return (
    <div key={entry.id} className="error-message">
      <span>{entry.message}</span>
      {entry.errorType === "config_error" && props.onOpenSettings && (
        <button className="error-settings-link" onClick={props.onOpenSettings}>
          Open Settings
        </button>
      )}
    </div>
  );
```

### 5. App.tsx: Wire `onOpenSettings` prop

**`claraity-vscode/webview-ui/src/App.tsx`** -- add prop to `<ChatHistory>`:
```tsx

```
