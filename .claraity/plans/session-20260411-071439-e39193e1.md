# Plan: Holistic Tool Schema Unification

## Problem Statement

The tool system has a structural flaw: **two competing sources of truth for tool schemas**, plus a latent bug that is masked by `**kwargs`.

### Current Architecture (broken)

```
tool_schemas.py          search_tools.py (_get_parameters)
─────────────────        ──────────────────────────────────
GREP_TOOL: file_path  ≠  GrepTool._get_parameters: path
GLOB_TOOL: path       ≠  GlobTool._get_parameters: path   <- matches, but only by accident
```

The LLM receives schemas from `Tool.get_schema()` → `_get_parameters()` (the implementation in search_tools.py).
`tool_schemas.py` is used for: subagent allowlisting, plan mode gating, MCP tool registry.

### Bugs Discovered

1. **GREP: parameter name mismatch** (the original issue)
   - `tool_schemas.py` GREP_TOOL says `file_path`
   - `search_tools.py GrepTool._get_parameters()` says `path`
   - `GrepTool.execute()` signature says `file_path` (correct)
   - Net effect: LLM sends `path=...`, Python receives it via `**kwargs`, silently ignored → grep always searches current directory

2. **GLOB: broken execute() body** (newly discovered, more severe)
   - `GlobTool.execute()` signature correctly says `path: str | None = None`
   - But the body references `file_path` (undefined variable) → `NameError` at runtime
   - This means `glob` with any path argument crashes. It only works when `path=None` (default ".")
   - `_get_parameters()` says `path` (matches signature, but body is wrong)

3. **GREP: stale path error message**
   - Line 266: `error=f"Path not found: {path}"` but the variable is `file_path` → always says `path=None` in error

4. **Design flaw: opt-in delegation pattern**
   - `ReadFileTool` correctly delegates `_get_parameters()` to `tool_schemas.py`
   - `GrepTool` and `GlobTool` do not — they have their own inline copies
   - Nothing enforces the delegation; future tools will repeat this mistake

### Why `**kwargs` masks bug #1

`GrepTool.execute()` accepts `**kwargs`. When the LLM sends `path="src/core"`, Python receives it in `**kwargs` (not as `file_path`), so `file_path` defaults to `None` and grep searches `.`. No exception is raised. The tool appears to work but silently ignores the path.

## Proposed Fix: Three-Layer Approach

### Layer 1: Fix the immediate bugs (search_tools.py)

**GrepTool:**
- `_get_parameters()`: delegate to `GREP_TOOL.parameters` from tool_schemas.py (like ReadFileTool does)
- `execute()`: already uses `file_path` — no change needed
- Fix error message line 266: `path` → `file_path`

**GlobTool:**
- `execute()` signature: rename `path` → `file_path` (matches tool_schemas.py intent)
- `execute()` body: fix all references from `file_path` (undefined) to the correct variable
- Fix error message line 642: `path` → `file_path`  
- `_get_parameters()`: delegate to `GLOB_TOOL.parameters` from tool_schemas.py

**GLOB_TOOL in tool_schemas.py:**
- Already has `path` — update to `file_path` to match all other file tools (this is what the other session intended)

### Layer 2: Enforce the pattern via base class

Add a class-level hook to `Tool` base class so delegation to `tool_schemas.py` is the default, not opt-in:

```python
# In Tool base class:
_SCHEMA_NAME: str | None = None  # Subclasses can set to tool name in tool_schemas

def _get_parameters(self) -> dict[str, Any]:
    if self._SCHEMA_NAME is not None:
        from src.tools.tool_schemas import _SCHEMA_REGISTRY
        return _SCHEMA_REGISTRY[self._SCHEMA_NAME].parameters
    raise NotImplementedError("Subclass must implement _get_parameters()")
```

And in tool_schemas.py, build a name→ToolDefinition registry:
```python
_SCHEMA_REGISTRY: dict[str, ToolDefinition] = {t.name: t for t in ALL_TOOLS}
```

Then tools that are in tool_schemas.py just set `_SCHEMA_NAME = "grep"` and get delegation for free. Tools NOT in tool_schemas.py (director tools, knowledge tools, background tools) keep their own `_get_parameters()`.

### Layer 3: Add a CI-level consistency check

Add a test that:
1. Instantiates every Tool subclass
2. Calls `get_schema()` on each
3. Asserts the schema matches the corresponding `ToolDefinition` in `tool_schemas.py` (if one exists)
4. Asserts parameter names in the schema match the `execute()` signature

This makes future divergence a test failure, not a silent bug.

## Execution Order

**Slice 1: Fix the live bugs in search_tools.py**  
Critical path — fixes the actual broken behavior. GlobTool NameError and GrepTool silent path ignore.

**Slice 2: Align tool_schemas.py (GLOB_TOOL path → file_path) + add _SCHEMA_REGISTRY**  
Prerequisite for Slice 3. Safe change — only affects subagent allowlisting description, not runtime behavior.

**Slice 3: Wire _get_parameters() delegation in GrepTool and GlobTool**  
Now that Slice 1 fixed the execute() signatures and Slice 2 aligned tool_schemas.py, delegation is safe.

**Slice 4: Base class _SCHEMA_NAME pattern**  
Structural improvement. Makes future tools use delegation by default.

**Slice 5: Consistency test**  
Locks in the fix permanently.

## Risks and Mitigations

| Risk | Mitigation |
|------|-----------|
| Renaming GlobTool param from `path` to `file_path` breaks callers | grep for all GlobTool instantiation/call sites; `**kwargs` in execute() catches unknown args gracefully |
| _SCHEMA_REGISTRY import creates circular import | Use lazy import (TYPE_CHECKING guard) same as other tool_schemas imports |
| Base class change breaks director/knowledge tools | Those tools have their own `_get_parameters()` — the base class only raises if `_SCHEMA_NAME` is None AND no override exists (current behavior) |
| Test brittleness | Test only checks tools that have a matching entry in tool_schemas.py by name — ignores tools that don't |
