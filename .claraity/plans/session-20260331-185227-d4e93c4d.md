# Fix: `claraity.newChat` Title Bar Icon Does Not Start a New Session

## Problem Statement

The `claraity.newChat` VS Code command (bound to the `+` icon in the native title bar) only
focuses the chat view and restarts a dead agent. When the agent is alive and connected it does
nothing — it never sends `new_session` to the Python server.

All four other title-bar icons follow the same pattern:
  command handler → sidebarProvider.showXxx() → postToWebview() → webview reacts

`newChat` was never updated to follow this pattern when the icons moved from the in-webview
status bar to the native VS Code title bar.

## Root Cause (confirmed)

`extension.ts:187-195` — the `else` branch (agent alive) has no body:

```ts
vscode.commands.registerCommand('claraity.newChat', () => {
    vscode.commands.executeCommand('claraity.chatView.focus');
    if (connection && !connection.isConnected) {
        connection.restart();
    }
    // nothing here when connected
}),
```

The actual `new_session` send path exists in `sidebar-provider.ts:469` but is only reachable
via a webview `newSession` message — which only fires from the SessionPanel's internal button,
not from the title bar icon.

## Decision: Direct `connection.send()` (not webview round-trip)

Two approaches exist:

**Option A — webview round-trip** (match other icons):
  command → sidebarProvider.newChat() → postToWebview({type:'showNewChat'}) → webview sends newSession → sidebar routes to new_session

**Option B — direct send** (simpler, more correct):
  command → connection.send({type:'new_session'})

Option B is chosen because:
- "New Chat" is a backend state reset, not a UI panel navigation. No webview state change is
  needed before the server responds.
- The server already sends back `session_info` on reset, which the webview handles at
  `App.tsx:57` — the webview will update reactively without being told to navigate.
- Adding a webview round-trip adds latency and a new message type for no benefit.
- `claraity.interrupt` already uses this same direct pattern (`connection?.send({type:'interrupt'})`).
  `newChat` is analogous — it's an agent lifecycle command, not a panel-switch command.

## Changes

### 1. `claraity-vscode/src/extension.ts` — Fix the command handler

Add `connection?.send({ type: 'new_session' })` in the connected branch.

Before:
```ts
vscode.commands.registerCommand('claraity.newChat', () => {
    vscode.commands.executeCommand('claraity.chatView.focus');
    if (connection && !connection.isConnected) {
        log.appendLine('[ClarAIty] Agent is dead on newChat — restarting');
        connection.restart();
    }
}),
```

After:
```ts
vscode.commands.registerCommand('claraity.newChat', () => {
    vscode.commands.executeCommand('claraity.chatView.focus');
    if (connection && !connection.isConnected) {
        log.appendLine('[ClarAIty] Agent is dead on newChat — restarting');
        connection.restart();
    } else {
        connection?.send({ type: 'new_session' });
    }
}),
```

### 2. `claraity-vscode/src/__tests__/extension.test.ts` — Update and add tests

**Update existing test** `'newChat command focuses the chat view'`:
- Currently only asserts `executeCommand('claraity.chatView.focus')`.
- Split into three focused tests:
  1. `'newChat always focuses the chat view'` — fires regardless of connection state
  2. `'newChat sends new_session when agent is connected'` — `isConnected=true` path
  3. `'newChat restarts agent when disconnected'` — `isConnected=false` path (existing behaviour)
  4. `'newChat does not send new_session when agent is disconnected'` — guard test

The connected-path tests require setting `mockStdioInstance.isConnected = true` before
invoking the callback so the `else` branch is exercised.
