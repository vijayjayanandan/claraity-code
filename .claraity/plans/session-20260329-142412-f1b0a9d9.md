# Plan: Stop API keys leaking into webview

## Summary
Three targeted changes to ensure API keys and search keys never enter the
webview's JavaScript state. The `has_api_key` / `has_search_key` boolean
flags are sufficient for the UI; the actual values are not needed there.

## Changes

### 1. `src/server/config_handler.py` — remove `api_key` from `config_loaded`
Line 76 currently sends `"api_key": cfg.api_key or ""`.
Remove that line. `has_api_key` (line 75) is already there and is all the
UI needs to show "(key stored)".

### 2. `claraity-vscode/src/sidebar-provider.ts` — remove search key injection
Line 254 currently does `config.search_key = tavilyKey` — injecting the
full Tavily key into the message sent to the webview.
Remove that line. Keep `config.has_search_key = true` (line 253).

### 3. `claraity-vscode/webview-ui/src/components/ConfigPanel.tsx` — keys are write-only
- Remove `apiKey` and `searchKey` from `ConfigSnapshot` and `snapshotEqual`
  (they're never received from the server anymore, so they can't dirty-track)
- Keep `hasApiKey` / `hasSearchKey` for the "(key stored)" label
- The password fields start empty; only send to server if the user typed something
- On save: only include `api_key` in the payload if the field is non-empty;
  same for `search_key`
- `parseSnapshot` no longer reads `api_key` or `search_key` from server data

### Result
- Keys never enter React state
- Keys never appear in any console log
- Dirty tracking still works (based on all other fields)
- "(key stored)" labels still show correctly via `has_api_key` / `has_search_key`
- A user can still update a key by typing a new one into the field and saving
