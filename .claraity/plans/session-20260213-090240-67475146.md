## Summary
Fix screenshot and file paste functionality (Ctrl+V) in the Textual TUI by using the existing `ClipboardHandler` class and fixing the event handling flow.

## Root Cause Analysis

### Key Finding: Textual's Paste Event Only Contains Text
From Textual documentation: "Event containing text that was pasted into the Textual application. This event will only appear when running in a terminal emulator that supports bracketed paste mode."

**Critical Insight:** The `Paste` event in Textual **only provides text** (`event.text`). It does NOT provide binary data like images or file lists. This is a fundamental limitation of terminal emulators.

### Current Implementation Issues

1. **ClipboardHandler exists but is NOT used**
   - File: `src/ui/clipboard_handler.py`
   - Has proper `get_clipboard_content()` method that returns `(image_bytes, file_list, text)`
   - Never imported or used in `src/ui/app.py`

2. **Timing issues with current approach**
   - `on_key()` intercepts Ctrl+V and schedules `_handle_paste()` asynchronously
   - Double-deferred scheduling (`call_next` → `create_task`) introduces race conditions
   - By the time PIL reads clipboard, data may be stale or inaccessible

3. **Silent failures**
   - Exceptions logged at `debug` level, not visible to users
   - No retry logic when PIL fails transiently
   - Falls back to text paste immediately, causing file paths to be pasted as text

### Why It Worked Before (Hypothesis)
The code likely worked in an earlier version that:
- Called PIL synchronously in the event handler
- Had immediate clipboard access without async deferral
- Or used a different event handling pattern

## Implementation Plan

### Approach: Use ClipboardHandler Synchronously in on_key()

Since we can't await in `on_key()`, we'll:
1. Call `ClipboardHandler.get_clipboard_content()` **synchronously** in `on_key()`
2. If we get image/files, schedule async attachment handlers
3. If we get text only, insert it directly (no async needed)
4. Prevent `on_paste()` from double-pasting

### Step 1: Import and Use ClipboardHandler

**File:** `src/ui/app.py`

Add import at top:
```python
from .clipboard_handler import ClipboardHandler
```

### Step 2: Rewrite on_key() Ctrl+V Handler

Replace current implementation with synchronous clipboard check:

```python
# Ctrl+V: Check clipboard synchronously BEFORE any async scheduling
if event.key == "ctrl+v":
    event.prevent_default()
    event.stop()
    
    # Suppress subsequent on_paste() for 200ms
    self._suppress_paste_text_until = time.time() + 0.2
    
    # Get clipboard content synchronously (no async/await needed)
    image_bytes, file_list, text = ClipboardHandler.get_clipboard_content()
    
    if image_bytes:
        # Schedule async image attachment
        self.call_next(lambda: asyncio.create_task(
            self._attach_image_bytes(image_bytes)
        ))
        return
    
    if file_list:
        # Schedule async file attachment
        self.call_next(lambda: asyncio.create_task(
            self._attach_clipboard_files(file_list)
        ))
        return
    
    # Text only - insert directly (synchronous, no scheduling needed)
    if text:
        self.insert(text)
    
    return
```

### Step 3: Create _attach_image_bytes() Method

New method to handle pre-converted image bytes:

```python
async def _attach_image_bytes(self, image_bytes: bytes) -> None:
    """Attach image from clipboard bytes."""
    try:
        att = self.attachments.add_screenshot(image_bytes, "png")
        self.app.notify(f"Attached image ({att.size_kb:.1f} KB)")
        self._update_attachment_indicator()
    except Exception as e:
        logger.error(f"Failed to attach image: {e}", exc_info=True)
        self.app.notify(f"Failed to attach image: {e}", severity="error")
```

### Step 4: Update _attach_clipboard_files()

Simplify to accept file list directly (no PIL calls):

```python
async def _attach_clipboard_files(self, files: list) -> None:
    """Attach files from clipboard file list."""
    from pathlib import Path
    valid_count = 0
    
    for file_path in files[:5]:  # Limit to 5 files
        if isinstance(file_path, str) and Path(file_path).exists() and Path(file_path).is_file():
            try:
                att = await self.attachments.add_file(file_path)
                if att.kind == "image":
                    self.app.notify(f"Attached image ({att.size_kb:.1f} KB)")
                else:
                    self.app.notify(f"Attached: {att.filename}")
                valid_count += 1
            except (ValueError, FileNotFoundError, IOError) as e:
                logger.error(f"Failed to attach {file_path}: {e}", exc_info=True)
                self.app.notify(f"Failed: {e}", severity="error")
    
    if valid_count == 0 and files:
        self.app.notify("No valid files found in clipboard", severity="warning")
    
    self._update_attachment_indicator()
```

### Step 5: Remove Obsolete Methods

Delete these methods (no longer needed):
- `_start_handle_paste()`
- `_handle_paste()`
- `_attach_clipboard_image()` (replaced by `_attach_image_bytes()`)

### Step 6: Keep on_paste() Guard

Keep the suppression logic in `on_paste()` to prevent double-paste:

```python
async def on_paste(self, event: Paste) -> None:
    """Handle paste events from terminal emulator (right-click paste, etc)."""
    # Check if we should suppress text paste (Ctrl+V handler just ran)
    if time.time() < self._suppress_paste_text_until:
        logger.debug("Suppressing on_paste() - Ctrl+V handler active")
        return
    
    # For terminal-initiated pastes, just insert the text
    if event.text:
        self.insert(event.text)
```

## Why This Will Work

1. **Synchronous clipboard access** - No timing issues, PIL reads clipboard immediately when Ctrl+V is pressed
2. **ClipboardHandler handles all edge cases** - Already has retry logic, format detection, error handling
3. **Single scheduling layer** - Only schedule async work when actually needed (file I/O)
4. **Text paste is synchronous** - No delay, no race conditions
5. **Proper error logging** - ClipboardHandler returns None on failure, we can log appropriately

## Trade-offs

**Pro:**
- Fixes both screenshot and file paste issues
- Uses existing, tested ClipboardHandler code
- Simpler flow, easier to debug
- No timing races

**Con:**
- `ClipboardHandler.get_clipboard_content()` does PIL calls synchronously in UI thread
- Could block UI for ~10-50ms on slow clipboard access
- Acceptable trade-off: clipboard access is usually fast, and blocking is only during Ctrl+V

## Alternatives Considered

### Alternative 1: Keep async approach with retry logic
- Add retry loop in `_handle_paste()`
- Still has timing issues, just reduces frequency
- More complex, harder to debug

### Alternative 2: Use threading for PIL calls
- Run `ImageGrab.grabclipboard()` in thread pool
- Adds complexity, still has scheduling overhead
- Not worth it for clipboard access (usually fast)

### Alternative 3: Poll clipboard on focus
- Check clipboard when input gains focus
- Doesn't help with Ctrl+V timing
- Adds unnecessary overhead

## Chosen Approach
**Use ClipboardHandler synchronously in on_key()** - simplest, most reliable, uses existing tested code.
